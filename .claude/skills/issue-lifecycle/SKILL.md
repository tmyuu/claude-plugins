# Issue ライフサイクルの原則

このスキルは **Hook で強制されない判断基準・原則** を記述する。
コミット・ブランチ・ラベル・ステータス遷移などの**機械的なルールは全て Hook で自動化・強制**されているため、ここには記載しない。

## Issue の書き方

### タイトル
- **クライアントが読むもの**として書く（技術用語を最小限に）
- 良い例: 「LINE連携: Webhook設定・UX改善」「顧客ダッシュボードに月次レポート追加」
- 悪い例: 「fix: webhook HMAC validation error」「refactor: extract useAuth hook」

### 本文
後から見返して経緯がわかるように:
- **背景**: なぜこの作業が必要か（ビジネス的な理由）
- **目的**: 何を達成するか（ユーザーにとっての価値）
- **完了条件**: 何をもって完了とするか（チェック可能な条件）
- **アクションアイテム**: チェックリスト形式で作業を分解

## 重複の回避

- 作成前に**既存のオープン Issue を確認**（SessionStart で注入される）
- 同じ目的の Issue が既にあれば新規作成しない
- 完了済みだが再対応が必要 → 新規 Issue を作成して旧 Issue を参照

## 親子関係（エピック / タスク）

**独立 Issue の乱立を避けるため、基本は親 Issue にぶら下げる。**

### 階層の考え方
- **親（エピック）**: ビジネス目的・機能テーマ・フェーズ単位
  - 例: 「LINE連携機能の実装」「v2 リリース」「リピーター率向上 第1フェーズ」
- **子（タスク）**: 独立した作業単位、親の完了条件の一部
  - 例: 「Webhook 設定」「署名検証」「UI 刷新」

### 判断基準
- 同じビジネス目的を共有 → **子 Issue**
- 同じフェーズの作業 → **子 Issue**
- 親 Issue の完了条件の一部を構成 → **子 Issue**
- 単発で完結し他と関連しない → **独立 Issue**（例: typo 修正、緊急障害対応、横断的な CI 設定）

### 運用
- 子 Issue の紐付けは **GitHub Sub-issues API** を使う
- 親 Issue のアクションアイテムにチェックリストとして子を追加
- 子 Issue のクローズと親のチェック更新は**セットで行う**（Hook で自動化）

## 依存関係

- 先に完了すべき Issue がある → **Blocks / Blocked by** を設定
  - 例: 実装 Issue → テスト Issue → 検収 Issue → 納品 Issue
- 重複 Issue は **Duplicates** でマークしクローズ

## 作業の分割判断

作業中に以下のいずれかに該当すると判断したら、サブ Issue に分割する:

- **独立した変更が複数ある**: 互いに依存しない複数の変更が必要
- **スコープが膨らんでいる**: 当初の想定を超える変更が必要になった
- **別の問題を発見した**: 今の Issue とは直接関係ない問題を見つけた

### 分割しない場合
- 変更が小さく、1コミットで完結する
- 変更同士が強く依存していて分割すると逆に複雑になる

## プロジェクト選択

- リポジトリにリンクされたプロジェクトのみ候補
- 既存プロジェクトのテーマに合うなら**必ず既存を使う**
- テーマが明らかに逸脱している場合のみ新規作成を検討
- 判断に迷う場合はユーザーに確認

## 検収

- Acceptance タイプの Issue は、進行中になるとクライアントに自動通知される
- 検収承認でクローズ、差し戻しで再オープン

## やってはいけないこと

- 確認せずに Issue をクローズ
- 既存プロジェクトを無視して新規作成
- 親子関係を `Parent: #N` テキストで表現（Sub-issues API を使う）
